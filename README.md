## Главные выводы
1. **Middleware управляет потоком запросов.** Цепочка `app.Use(...)` выполняется строго в порядке объявления. Это позволяет централизованно решать задачи логирования, валидации, авторизации и добавления заголовков до попадания запроса к конечному обработчику.
2. **Minimal API ускоряет разработку.** В отличие от классического подхода, не требуется создавать отдельные классы-контроллеры. Вся логика маршрутизации пишется компактно в `Program.cs`, что идеально для микросервисов и лёгких API.
3. **Автоматическая сериализация упрощает работу с фронтендом.** ASP.NET Core автоматически конвертирует C#-объекты в JSON, приводя имена свойств к `camelCase`. Это соответствует стандартам JavaScript и избавляет от ручного форматирования ответов.

## Ответы на контрольные вопросы

**1. Чем Minimal API отличается от MVC в ASP.NET Core?**  
`Minimal API` использует декларативный подход в одном файле `Program.cs` с помощью методов `MapGet/MapPost`, не требуя создания отдельных классов-контроллеров, представлений (Views) или сложной конфигурации маршрутов. `MVC` (Model-View-Controller) подразумевает разделение логики на контроллеры, модели и представления (Razor), что удобнее для крупных приложений с серверным рендерингом страниц и сложной бизнес-логикой.

**2. Что произойдёт, если поставить `app.MapGet(...)` до `app.Use(...)`?**  
Запрос будет обработан маршрутом и сразу вернёт ответ, не доходя до зарегистрированного ниже `middleware`. Поскольку `Middleware` работает как конвейер, порядок регистрации критичен: если маршрут стоит первым, он "обрывает" цепочку, и последующие `Use` (например, логирование, проверка ключей или заголовки) не выполнятся для этого запроса.

**3. Почему ASP.NET Core преобразует PascalCase в camelCase при сериализации JSON?**  
Это сделано для соответствия стандартам экосистемы веб-разработки. В C# принято использовать `PascalCase` для имён свойств и методов, тогда как в JavaScript и спецификации JSON стандартом является `camelCase`. ASP.NET Core по умолчанию использует `JsonNamingPolicy.CamelCase`, чтобы сгенерированный JSON сразу был готов к использованию на фронтенде без дополнительного преобразования ключей.

**4. Что означает код ответа HTTP 401? А 404? А 200?**  
- `200 OK` — Запрос успешно обработан, сервер вернул ожидаемые данные.
- `401 Unauthorized` — Запрос требует аутентификации. Клиент не предоставил учётные данные или они неверны.
- `404 Not Found` — Сервер не нашёл запрошенный ресурс по указанному URL (маршрут не настроен или адрес введён с ошибкой).

**5. Чем `dotnet run` отличается от `dotnet watch run`?**  
`dotnet run` компилирует проект и запускает сервер один раз. Для применения изменений код нужно остановить и запустить заново. `dotnet watch run` запускает приложение в режиме наблюдения за файловой системой: при любом сохранении изменений в исходном коде он автоматически перекомпилирует проект и перезапускает сервер, что значительно ускоряет процесс разработки.
